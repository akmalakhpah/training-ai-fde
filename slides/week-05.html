Retrieval and Grounding (RAG)

A model only knows what it was trained on and what you put in the prompt. RAG is how you put the right thing in the prompt.

The failure mode

'The model was fluent, authoritative, and completely wrong — and no one in the room could tell.'

— the hallucination problem, in one sentence

Why grounding matters

Memory vs retrieved evidence

Ungrounded model

  • Answers from training weights
  • Knowledge has a cutoff date
  • Cannot cite a source
  • Confident when wrong
  • Fails silently — no warning

Grounded with RAG

  • Answers from retrieved source text
  • Knowledge as fresh as your corpus
  • Can cite the chunk it used
  • Can say 'not in the documents'
  • Fails visibly — you can catch it
The pipeline

RAG end to end — two phases

Phase 1: Indexing (once)

  1. Split documents into chunks
  2. Embed each chunk into a vector
  3. Store vectors in a vector store

Phase 2: Query (every request)

  1. Embed the user question
  2. Retrieve the top-k nearest chunks
  3. Assemble context + question into a prompt
  4. Call the model — answer from context only
Indexing — step 1

Chunking strategies and tradeoffs

  • Fixed-size: every chunk is N tokens. Fast and predictable, but may split mid-sentence.
  • Paragraph-based: split on blank lines. Respects natural boundaries, produces variable-size chunks.
  • Heading-based: one chunk per section. Works well for structured documents like syllabi or manuals.
  • With overlap: repeat the last K tokens in the next chunk — prevents answers from being split at a boundary.
  • Working default: 256 to 512 tokens with 10 to 15% overlap. Start here, measure, then adjust.
Indexing — step 2

Embeddings and similarity search

  • An embedding converts text into a vector that encodes meaning, not keywords.
  • Similar meaning produces nearby vectors; dissimilar meaning produces distant vectors.
  • Cosine similarity is the standard metric: closer to 1.0 means more similar.
  • A vector store (e.g. pgvector, Chroma, Pinecone) holds vectors and answers nearest-neighbour queries efficiently.
  • Top-k retrieval: return the k chunks whose vectors are closest to the query vector.
Query phase

Assembling the grounded answer

Context: [chunk 1 text] [chunk 2 text] [chunk 3 text] Question: {user question} Answer using only the context above. If the answer is not in the context, say: 'I cannot find this in the provided documents.'
  • Retrieved chunks go first — the model reads them as evidence before the question.
  • The fallback instruction is your primary safeguard against hallucination on missed retrieval.
  • Include the source reference in the answer so users can verify the claim.
Live · ~14 min

RAG working well — then confidently wrong

  • Query 1 (works): ask something with a clear, focused answer in one chunk. Show the similarity score, the retrieved chunk, the grounded answer, and the source citation.
  • Query 2 (fails): ask something whose answer is split across a chunk boundary, or whose keyword matches an irrelevant section. The wrong chunk scores highest.
  • Observe: the model generates a confident, fluent, wrong answer. No error. No warning. Just wrong.
  • Diagnose together: bad chunk boundary? Ambiguous query term? Semantic-but-irrelevant match?
Failure modes

When RAG retrieves the wrong thing — and why

  • Missing context: the right chunk was never indexed, or dissolved into a larger chunk that dilutes its retrieval signal.
  • Distracting context: a semantically similar but factually irrelevant chunk scores higher than the correct one.
  • Split answer: the answer spans a chunk boundary — neither half alone is sufficient to answer the question correctly.
  • Ambiguous query: a short or polysemous question matches the wrong part of the corpus by vector similarity.
  • The scary failure: the model does not say 'I don't know' — it answers fluently from the wrong evidence.
Tool selection

When RAG is the wrong tool

Use RAG when

  • Corpus is too large for one prompt
  • Questions are varied and semantic
  • Corpus changes frequently
  • Source citation is required
  • Precision of retrieval can be measured

Skip RAG when

  • A DB lookup returns the exact answer
  • The corpus fits in a single prompt
  • A simple rule or keyword filter is enough
  • Latency budget is very tight
  • Corpus is too small to chunk meaningfully
Evaluating the system

Measuring retrieval quality

  • Write five questions with known correct source chunks before you start building.
  • Run retrieval for each question; check whether the correct chunk appears in the top-k results.
  • Precision at k: what fraction of your test queries return the right chunk in position 1 through k?
  • Use this score to compare chunking strategies — it is your north-star metric for retrieval.
  • A retrieval eval prevents you from shipping a 'feels good' pipeline that is actually broken on edge cases.
Required: reliability check

Flagging low-confidence answers

  • Identify one case in your corpus where wrong retrieval could directly mislead a user.
  • A similarity threshold is the simplest signal: if the top match scores below your cutoff, surface a warning rather than an answer.
  • The warning is not 'something went wrong' — it is: 'I could not find a confident answer in the provided documents.'
  • Include this safeguard in your submission — it is explicitly weighted in the assessment rubric.
  • Stretch: surface the similarity score alongside every answer so users can judge confidence themselves.
This week's project

Build a RAG system over a real corpus

  • Build: chunk, embed, store, retrieve, and generate a grounded answer over a real document set (past papers, syllabus, or similar).
  • Show it working: at least one query where retrieval clearly returns the right chunk with a high similarity score.
  • Show it failing: at least one genuine failure with your honest explanation of the cause.
  • Write a failure analysis: where it works, where it fails, and what you would change next.
  • Reliability check (required): one case where wrong retrieval could mislead a user, and how you flag low-confidence answers.
Self-learn (6–8 hours)

How to spend your self-learn time

  1. Topic A (2–2.5 hrs): Build RAG from scratch — follow a tutorial end to end on a small corpus before touching your own. Do this first.
  2. Topic B (1.5 hrs): Chunking strategies — chunk the same document three ways and compare what retrieves for one question.
  3. Topic C (1.5 hrs): Embeddings and measurement — write five questions with known correct sources and measure precision at k.
  4. Topic D (45 min): When not to use RAG — name one task in your corpus where RAG is overkill and explain why.
Don't

Common pitfalls this week

  • Picking a chunk size at random and never measuring whether it actually helps retrieval.
  • Skipping the fallback instruction — without it the model hallucinates from memory whenever retrieval misses.
  • Only demonstrating the happy path in your project submission — the failure analysis is 30% of the rubric.
  • Treating retrieval failure as an infrastructure problem — most failures are chunking strategy and query design issues.
  • Reaching for RAG on a task that a simple database lookup handles better, cheaper, and faster.
Recap

If you remember three things

  • RAG grounds the model in retrieved evidence — but wrong retrieval produces confident wrong answers, which is worse than a refusal.
  • Chunk size, overlap, and embedding choice all affect retrieval quality — measure with a small labelled eval set before you ship anything.
  • RAG is not always the right tool — a database lookup, a plain prompt, or a simple rule often wins on speed and simplicity.
