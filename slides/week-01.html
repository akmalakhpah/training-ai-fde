Week 01 · Phase 1 — Production Foundations

From Coursework to Production

Most software does not fail in development — it fails in deployment. This week is about the habits that survive contact with a real system.

Taught class 90 min · self-learn 6–8 hrs · project 6–8 hrs · assessment 2–3 hrs.

The gap

'It works on my machine.'

— every engineer, right before production broke

Why this whole course exists

The four muscles

  • Ship real work — produce production code others depend on, not coursework that runs once.
  • Stay functional in broken systems — stay calm and keep moving inside ambiguous, half-broken, incomplete systems.
  • Translate between humans and machines — turn a vague human problem into a technical plan, a technical tradeoff back into plain language, and defend the call to a customer.
  • Supervise the AI agents — brief, scope, review, and guardrail the agents that do the work; never blindly trust them.

This week leans hardest on shipping real work — but every week trains all four at once.

Mental model shift

Two very different worlds

Student repo

  • One author
  • Runs once, on your laptop
  • No tests, or tests run once
  • No review, straight to main
  • 'Done' = the demo runs
  • Deleted after grading

Production system

  • Many authors over years
  • Local, staging, prod — always running
  • Tests gate every single merge
  • Every change reviewed before merge
  • 'Done' = merged, tested, reviewed, observable
  • Lives until the business decides otherwise
Live tour

Anatomy of a production repo

  • Top-level structure: where config lives, where source lives, where tests live.
  • The README: how to run it, how to test it, who owns it.
  • The CI config: what runs on every push, what blocks a merge.
  • A CLAUDE.md: project context for AI-assisted development.
  • Signals of health: tests folder, clear module boundaries, no orphaned files.
  • Signals of trouble: no tests, one giant file, README that lies.
Survival skill

Reading a codebase you did not write

  • Start at the entry point — where does this thing actually run?
  • Follow one request: entry → business logic → database → response. Trace it live.
  • Use jump-to-definition, not keyword search. Follow the call stack down.
  • Read the tests first — they tell you what the code is supposed to do.
  • Ignore error handling on the first pass. Understand the happy path first.

GitHub for real

Beyond commit and push

  • Feature branches: one branch per concern, named for the change, short-lived.
  • Rebase vs merge: rebase cleans up your own branch; merge preserves shared history.
  • When to rebase: before opening a PR, to replay your commits on top of main.
  • When NOT to rebase: any branch others are already working on.
  • Merge conflicts are normal — git is asking you to make a decision humans must make.

Live exercise: create a branch, commit twice, manufacture a conflict, resolve it from the CLI.

The shipping loop

The pull request lifecycle

  1. Cut a branch off main for one focused change.
  2. Open a PR — even for small changes. Description: why, what, how to test.
  3. Request review. Address every comment thoughtfully, not defensively.
  4. Merge only when CI is green and reviewers have approved.
  5. Watch the deploy. The job is not done until it is running in production.

What a good PR looks like: small, one concern, clear description, tests included.

Live · ~12 min

Walking through a real PR

  • Open a real PR on the training repo — what is this change and why does it exist?
  • Read the description: why now, what changed, how was it tested?
  • Read the review thread — multiple rounds of comments, author responding well.
  • Open the diff tab: is this one concern or three different things?
  • Watch CI run: which check failed, what did the log say, how was it fixed?
The automated gate

CI/CD with GitHub Actions

On every push, CI runs

  • Lint and format checks
  • Type checks
  • Unit and integration tests
  • Build the artifact
  • Sometimes: deploy to staging

When CI is red

  • Do not re-run hoping it passes
  • Open the failing job log
  • Find the actual error, not just the red X
  • Fix the cause, then push
  • Can't reproduce locally? That gap is the bug
Shared code needs tests

Testing code you did not write

  • Tests are how you change things without fear in a shared codebase.
  • Before touching anything: run the suite. Know your baseline.
  • Write a happy-path test: does the function do what it claims when inputs are valid?
  • Write an edge-case test: empty input, zero, null, boundary value.
  • A test that cannot fail tells you nothing. Ask: what would break this?

Live: pick one untested function together. Write both tests on screen. Run them. Watch one fail before the fix, pass after.

Live · ~12 min

Driving one Claude Code agent

  • Brief it with context: module name, specific task, explicit scope constraint.
  • Scope the task narrowly — one function, not the whole file.
  • Review the diff line by line before accepting. Read every changed line.
  • Show it being confidently wrong: ask it to justify a suspicious change.
  • Run the tests. A passing diff that breaks a test is a rejected diff.

The agent is fast, not trustworthy. Your job is supervision.

Non-negotiable rep

The by-hand bug — why it matters

  • You cannot supervise an agent doing something you have never done yourself.
  • The by-hand fix is calibration — it builds the reference point for catching agent errors.
  • Method: read the full stack trace, find the file and line, open it, read context, hypothesise, test.
  • If you cannot explain the root cause in one sentence, you have not fixed it — you have hidden it.
  • Only after fixing it by hand do you then pick the second bug and brief Claude Code.

The agent is faster. You being able to catch its mistakes makes the combination safe.

Self-learn — 6 to 8 hrs

Four topics this week

A — GitHub beyond commit and push

Branch, rebase, and resolve a merge conflict without panic. Video + Atlassian guide + one AI-guided practice session.

B — Reading a large codebase

Orient in a repo you did not write. Answer the three diagnostic questions in writing: entry, database, response.

C — Claude Code fundamentals

Brief, run, and review a single agent. Verify its explanation against actual code line by line. Catch at least one mistake.

D — Testing code you did not write

Add a happy-path test and an edge-case test to an existing function. Make each test meaningful: it must be able to fail.

Self-learn A · 1.5–2 hrs

GitHub beyond commit and push

  • Goal: branch, rebase, and resolve a merge conflict without panic.
  • Start here: in the training repo, create a branch, make two commits, then manufacture a merge conflict against another branch and resolve it from the command line.
  • Watch: Git Branching Strategy & Git REBASE to fix Merge Conflicts (fallback search: "git rebase resolve merge conflicts tutorial").
  • Read: Atlassian Merging vs Rebasing · Resolve merge conflicts
Ask AI — paste into Claude

"I am new to git rebase. Walk me through, step by step, how to rebase my feature branch onto main, what to do when I hit a conflict, and how to tell whether I should rebase or merge in a shared repo. Then give me three practice exercises I can run locally to build the muscle."

Self-learn B · 1.5–2 hrs

Reading a large unfamiliar codebase

  • Goal: orient in a repo you did not write and find where to make a change.
  • Start here: open the training repo and answer in writing — where does a request enter, where does it hit the database, where does the response leave.
  • Read: Sparkbox How to Understand a Large Codebase · DEV How to Learn Unfamiliar Codebases
Ask AI — paste into Claude Code, inside the repo

"You are helping me onboard to this codebase. Without changing any files, give me a map: the entry points, the main modules and what each is responsible for, where data is read and written, and the path a single user request takes from entry to response. Then point me at three small, low-risk files I could read first to understand the conventions."

Self-learn C · 1–1.5 hrs

Claude Code fundamentals

  • Goal: brief, run, and review a single agent and catch its mistakes.
  • Start here: ask Claude Code to explain one module, then verify its explanation against the actual code line by line.
  • Read: Claude Code Quickstart — skim the whole page, then read the sections on CLAUDE.md and reviewing changes.
Ask AI — paste into Claude Code

"Explain what this module does and how it connects to the rest of the system. Cite the specific functions and line ranges you are basing each claim on, so I can verify you. Flag anything you are unsure about rather than guessing."

Self-learn D · 1–1.5 hrs

Writing tests for code you did not write

  • Goal: add a happy-path test and an edge-case test to existing code.
  • Start here: pick one untested function in the repo and write two tests for it.
  • Read: the testing section of the repo's README, plus your language's standard test runner docs (for example pytest or Jest).
Ask AI — paste into Claude

"Here is a function I did not write: [paste]. Help me understand its behaviour, then propose a happy-path test and one edge-case test. Do not write code that just mirrors the implementation. Explain what each test actually proves and what would make it fail."

This week's ship

The weekly project

  1. Clone the internal training repo. Install. Run the test suite. Note the baseline.
  2. Find a real bug. Fix it by hand, no agent. Note what the trace taught you.
  3. Find a second issue. Brief Claude Code. Review and accept the diff. Confirm tests pass.
  4. Add at least one test covering your change.
  5. Open one clean PR: clear title, why-focused description, tests, CI green.
  6. Get it through review and merged. Done = merged, not opened.

Deliverables: the merged PR link + a half-page written note on what the by-hand bug taught you.

Project · must-haves

Requirements checklist

  • Clone the internal training repo and get it running locally.
  • Find a first small, genuine bug and fix it by hand, no agent. Note what the stack trace or error told you.
  • Find a second genuine issue and fix it with Claude Code, reviewing the diff before accepting.
  • Add at least one test that covers your change.
  • Open one clean pull request: a clear title, a description of the change, and the tests.
  • Get the PR through review and merged, with CI passing.
Project · how to approach it

Suggested steps & stretch

  1. Clone, install, run the test suite, confirm it is green before you touch anything.
  2. Orient using the Topic B method. Write the three-question answer.
  3. Reproduce the first bug, read the trace, fix it by hand, add or update a test.
  4. Pick the second issue, brief Claude Code, review and accept its diff, confirm tests pass.
  5. Branch, commit with clean messages, push, open the PR, respond to review, merge.
Stretch (optional)
  • Add a second test that would have caught the bug before it shipped.
  • Leave a constructive review comment on a peer's PR.

Don't

Common pitfalls this week

  • Refactoring the whole file when you only need to fix one line — keep PRs small and focused.
  • Ignoring the codebase's existing style — match conventions, do not invent your own.
  • Pushing without running tests locally first — CI failure in public is avoidable.
  • Accepting a Claude Code diff you do not fully understand — you own every line you merge.
  • Getting defensive in review — comments are about the code, not about you.
  • Opening a PR without a clear description of why — reviewers are not mind readers.
Recap

If you remember three things

  • Production is a workflow, not a finished state — merged, tested, reviewed, observable.
  • Fix the first bug by hand. You cannot supervise an agent doing something you have never done.
  • Reading code is a skill: entry point, follow one request, answer the three questions.
