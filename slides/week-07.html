The landscape

Six kinds of automation — pick by shape of problem

  • Ask in the moment — Claude chat, Projects.
    One-offs. No setup, no reliability.
  • Scheduled assistant — Claude scheduled tasks, Cowork.
    Recurring judgement, low volume, zero infra.
  • No-code workflow — Zapier, Make, N8N, Activepieces.
    SaaS glue, event-driven, demo-able. ← this week
  • Code-first workflow — Windmill, Trigger.dev, Inngest, Temporal.
    Needs tests, versioning, real volume.
  • Custom code + cron — a script on GitHub Actions or Lambda.
    The forgotten answer. Often the right one.
  • Agent framework — Claude Agent SDK, MCP + agent loop.
    The task genuinely needs tools and reasoning. ← Week 6

Most real deployments are two or three of these together. Yours this week: a no-code workflow triggering an agent.

So why N8N?

Our vehicle for the week — and why

  • Visual — the customer can read the diagram. Most people paying for your work are not engineers.
  • Self-hostable — it runs inside a customer's environment when their data cannot leave it.
  • Fast — minutes to a working flow. On a deployment, speed to a demo is the real constraint.
  • It has MCP nodes — so last week's Anki agent plugs straight in.
  • Every tool has an exit condition. By the end of today you will name N8N's — and it is graded.
What N8N is

Nodes, triggers, actions, and data

The anatomy of a flow

  • Trigger node — starts the flow (webhook, schedule, event)
  • Action node — does one thing (HTTP call, Slack message, Sheet row)
  • Data — JSON objects that flow between nodes
  • Edge — the wire connecting two nodes

The rule

  • N8N is glue and orchestration
  • Real logic lives in your services, not in Function nodes
  • The flow should read like a sentence: 'when X, call Y, then deliver Z'
  • If a node grows past ~20 lines of JS, extract it
Block 3 · Triggers

Events and schedules, not manual clicks

Three trigger types

  • Manual — for development only; you click 'Execute'
  • Schedule — cron syntax; runs on a timer regardless of events
  • Webhook — any HTTP POST fires the flow immediately

Why it matters

  • Webhook = 'something happened, react now'
  • Schedule = 'every morning at 8am, do this'
  • Together they cover both event-driven and time-driven needs
  • Neither requires you to be awake or even online
Live · ~12 min

A first real flow: webhook to result

  • Add a Webhook trigger — N8N generates a unique URL automatically.
  • Fire a POST with a JSON body from curl or Postman.
  • Inspect the incoming data in the execution log — see the raw JSON.
  • Add a Slack or Google Sheets node — map one field from the payload to the destination.
  • Fire again. Watch the message arrive. No one clicked anything.
Live · ~12 min

Adding an AI call: Claude API inside the flow

  • Add an HTTP Request node after the webhook trigger.
  • Configure it as a POST to the Claude API — set the Authorization header and JSON body.
  • Map the incoming webhook payload as the user message field.
  • Inspect the Claude response — extract the text from the nested JSON path.
  • Wire the output to Google Sheets or Slack. Flow is now: event — AI — delivery.

Block 7 · Failure handling

Design the unhappy path on purpose

Silent failure is the worst failure

  • No visible error means no one knows work stopped
  • Work quietly not getting done for hours or days
  • Usually discovered by a customer, not by you
  • Worse than having no automation at all

Three layers of protection

  • Retry — handle transient errors automatically
  • Error branch — route failures to a fallback path or alert node
  • Alert — make failure visible in Slack or email immediately
The judgement call

Where no-code ends and code begins

Glue — stays in N8N

  • The trigger — webhook, schedule, new row
  • Routing and branching between services
  • Field mapping between systems
  • Delivery — Slack, Sheets, a database write

Logic — moves to code

  • Decisions and business rules
  • Prompts and anything schema-shaped
  • Anything that needs tests or version control
  • Anything a Function node grew past ~20 lines to do

The question that settles it: what breaks first if this runs 100×/minute instead of 10×/day?

Name them when the customer asks 'why N8N?'

What you'd move to — and when

  • Windmill — the closest upgrade. Open source, self-hosted, scripts-first with a generated UI. Real code, but you keep the diagram.
  • Trigger.dev / Inngest — code-first durable workflows. Proper retries and idempotency; you lose the visual the customer can point at.
  • Zapier / Make — only if the customer already lives there. Weaker for engineers, no self-hosting.
  • A cron job and 40 lines of Python — often the honest answer. Don't deploy a platform for a nightly task.
  • Licence check: N8N is Sustainable Use License, not open source. Deploy it freely; don't embed it in a product you sell.
Required submission component

Reliability check

One line in your submission. Answer both parts:

  • Name the step most likely to fail — be specific: which node, which failure mode.
  • Show what your flow does when it does — retry count, error branch path, alert destination.
  • 'It will not fail' is not an acceptable answer. Every external call can fail.
  • Test it: break that step on purpose before submitting. Include a screenshot of the error branch firing.
Required submission component · new this week

Boundary note

Two or three lines. The reliability check asks what happens when it breaks — this asks when it should stop being a flow at all.

  • Which step would you move out of N8N and into code first?
  • What forces the move — volume, testing, versioning, or logic that outgrew a node?
  • What would you move it to — your MCP server, your API, Windmill, a cron job?
  • 'Nothing' is not an answer. Every workflow has one. Naming it is the skill.

Don't

Common pitfalls this week

  • Hiding all logic in one giant Function node — N8N is orchestration, not a code editor.
  • Submitting a happy-path-only flow with no failure handling — it will be returned.
  • Using a Manual trigger in the final submission — external trigger is non-negotiable.
  • Tightly coupling the MCP server address into N8N so it breaks if the server moves.
  • Rewriting prompt logic inside the flow when the MCP agent already handles it.
  • Answering the boundary note with 'nothing' — every workflow has a step that outgrows the canvas.
