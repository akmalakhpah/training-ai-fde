What N8N is

Nodes, triggers, actions, and data

The anatomy of a flow

  • Trigger node — starts the flow (webhook, schedule, event)
  • Action node — does one thing (HTTP call, Slack message, Sheet row)
  • Data — JSON objects that flow between nodes
  • Edge — the wire connecting two nodes

The rule

  • N8N is glue and orchestration
  • Real logic lives in your services, not in Function nodes
  • The flow should read like a sentence: 'when X, call Y, then deliver Z'
  • If a node grows past ~20 lines of JS, extract it
Block 3 · Triggers

Events and schedules, not manual clicks

Three trigger types

  • Manual — for development only; you click 'Execute'
  • Schedule — cron syntax; runs on a timer regardless of events
  • Webhook — any HTTP POST fires the flow immediately

Why it matters

  • Webhook = 'something happened, react now'
  • Schedule = 'every morning at 8am, do this'
  • Together they cover both event-driven and time-driven needs
  • Neither requires you to be awake or even online
Live · ~12 min

A first real flow: webhook to result

  • Add a Webhook trigger — N8N generates a unique URL automatically.
  • Fire a POST with a JSON body from curl or Postman.
  • Inspect the incoming data in the execution log — see the raw JSON.
  • Add a Slack or Google Sheets node — map one field from the payload to the destination.
  • Fire again. Watch the message arrive. No one clicked anything.
Live · ~12 min

Adding an AI call: Claude API inside the flow

  • Add an HTTP Request node after the webhook trigger.
  • Configure it as a POST to the Claude API — set the Authorization header and JSON body.
  • Map the incoming webhook payload as the user message field.
  • Inspect the Claude response — extract the text from the nested JSON path.
  • Wire the output to Google Sheets or Slack. Flow is now: event — AI — delivery.
The two-week pipeline

Week 6 + Week 7 = one production system

What Week 6 gave you

  • An MCP server that runs a multi-step task using tools
  • Triggered manually from a terminal or client
  • Smart — it calls tools and reasons over results
  • Powerful but still requires a human to start it

What Week 7 adds

  • An external event fires N8N automatically
  • N8N calls your MCP server via HTTP Request
  • The result is routed to a useful destination
  • No manual step anywhere in the chain
Live · ~12 min

Wire the Week 6 MCP agent in — then break it on purpose

  • Add an HTTP Request node that calls the Week 6 MCP server.
  • Fire the full pipeline: external event — N8N — MCP agent — delivery.
  • Now break it: shut down the MCP server or point to a bad URL. Fire the flow again.
  • Show what silent failure looks like — the flow turns red and no one knows.
  • Add a retry, an error branch, and a Slack alert. Fire again. See the difference.
Block 7 · Failure handling

Design the unhappy path on purpose

Silent failure is the worst failure

  • No visible error means no one knows work stopped
  • Work quietly not getting done for hours or days
  • Usually discovered by a customer, not by you
  • Worse than having no automation at all

Three layers of protection

  • Retry — handle transient errors automatically
  • Error branch — route failures to a fallback path or alert node
  • Alert — make failure visible in Slack or email immediately
Required submission component

Reliability check

One line in your submission. Answer both parts:

  • Name the step most likely to fail — be specific: which node, which failure mode.
  • Show what your flow does when it does — retry count, error branch path, alert destination.
  • 'It will not fail' is not an acceptable answer. Every external call can fail.
  • Test it: break that step on purpose before submitting. Include a screenshot of the error branch firing.
Weekly project

Automate a real manual task end to end

Four requirements

  1. External trigger — webhook, new row, or incoming message (not a manual click)
  2. AI step — your Week 6 MCP agent or a Claude API call
  3. Useful delivery — Google Sheet, Slack message, or database row
  4. At least one failure path handled gracefully, not silently

Deliverables

  • Exported N8N workflow file
  • Short note on the manual task removed and why it matters
  • The reliability check (one line, both parts answered)
  • 3–5 min demo triggering the automation live and showing the failure path

Don't

Common pitfalls this week

  • Hiding all logic in one giant Function node — N8N is orchestration, not a code editor.
  • Submitting a happy-path-only flow with no failure handling — it will be returned.
  • Using a Manual trigger in the final submission — external trigger is non-negotiable.
  • Tightly coupling the MCP server address into N8N so it breaks if the server moves.
  • Rewriting prompt logic inside the flow when the MCP agent already handles it.
