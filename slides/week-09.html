Customer reality

Why customer environments are messy

What the demo assumes

  • Clean REST API with full docs
  • Sandbox environment available
  • Predictable, stable data shapes
  • Fast, reliable responses
  • Full read and write access
  • Vendor support is responsive

What you actually find

  • Docs last updated in 2018
  • No sandbox — prod only, be careful
  • Fields that are always null
  • Rate limits not mentioned anywhere
  • Read-only access, no exceptions
  • Support tickets take weeks
The FDE stance

Adapt to the customer's reality

  • The customer's systems are not wrong — they are the environment you must work in.
  • Your product must fit their world, not the other way around.
  • Constraints are requirements. Treat them as inputs to design, not complaints to escalate.
  • The FDE who says 'I can work with that' closes more projects than the one who says 'they should fix their system.'
Decision tree

Integration patterns for messy access

  1. Clean API: the best case — use it, respect rate limits, handle errors explicitly.
  2. Webhook: the system can push events to you — set up a receiver and verify signatures.
  3. Scheduled export: CSV, JSON, or XML on a timer — poll and process with deduplication.
  4. File drop: vendor puts files in FTP or S3 — watch the folder, process on arrival.
  5. UI automation: automate the browser or parse emails — document the fragility clearly.
  6. Scraping (last resort): fragile, vendor-dependent, legally grey — use only when nothing else exists.
Security and access

Authentication patterns

  • API key: a bearer token passed in a header or query param. Never commit it to git. Store it in env vars or a secrets manager. Rotate on suspected leak.
  • OAuth 2.0: delegated access without sharing passwords. The consent flow hands the user back a scoped token. Do not store the token on the client side.
  • JWT: a signed, self-contained token. Verify the signature server-side on every request. Check the expiry claim. Refresh before it expires, not after.
  • Webhook + HMAC signature: the sending system signs the payload with a shared secret. You recompute and compare the signature before processing. Skip this and anyone can spoof your endpoint.
Auth in two directions

Outbound vs inbound authentication

Outbound: you call them

  • API key in Authorization header
  • OAuth token for user-delegated access
  • JWT for service-to-service calls
  • Secret stored in env, never in code
  • Rotate keys on any suspected leak

Inbound: they call you (webhook)

  • External system sends a POST to your URL
  • Sender signs the payload with HMAC-SHA256
  • You recompute and compare signatures
  • Reject any request that does not match
  • Log every rejection — never drop it silently
Live · ~16 min

A real integration, warts and all

  • N8N as glue between two systems not designed to talk. Walk the full flow end to end.
  • Workaround 1: the field that is always null — add a fallback default, document why.
  • Workaround 2: undocumented rate limit at 100 req/min — add a delay and retry node.
  • Workaround 3: timezone offset is wrong in source data — transform timestamp before sending.
  • Workaround 4: doc says field exists, API returns 404 — add error handler and alert.
  • Auth: outbound call uses an API key; inbound webhook verifies the HMAC signature.
Working within constraints

Legacy constraints and partial access

  • Read-only access: pull data out, push it somewhere you control, transform it there.
  • No sandbox: make all test calls idempotent. Use a test record — never live customer data.
  • System you cannot change: build your adapter to absorb its quirks, not to fix them upstream.
  • Defensive integration: assume the other side will break. Add health checks, error alerts, and dead-letter handling.
  • When it breaks: your system must fail loudly, not silently — log and alert on every unexpected response shape.
Knowledge transfer

Documenting workarounds: the three-line note

# Workaround: fallback when vendor contact_email is always null # Why: vendor API v2 omits email for accounts created before 2021 (vendor support ticket #8823). # Watch: re-check after vendor migrates legacy accounts (target Q3). Remove fallback once field is populated. email = record.get('contact_email') or lookup_from_crm(record['id'])
  • Every workaround gets a note: what it does, why it was necessary, what to watch or revisit.
  • The next engineer inherits your workarounds. Undocumented, they look like bugs and get deleted.

Weekly project

Build and document a real integration

  • Connect two systems not designed to talk using N8N, a webhook, or custom code.
  • Example: push an AI step result into a Google Sheet, Slack, or a database, triggered by an external event.
  • At least one real authentication step: API key, OAuth, JWT, or verified webhook signature.
  • Every workaround documented with the three-line note (what, why, what to watch).
  • Evidence you adapted to constraints rather than wishing them away.
Project steps

How to approach the build

  1. Pick two systems and identify the data that must move between them.
  2. Decide the integration pattern and the auth method before writing a single line of code.
  3. Build the happy path first. Confirm data moves correctly end to end.
  4. Hit the inevitable snag. Build the workaround and write the three-line note immediately.
  5. Review the workaround documentation: could a stranger maintain this without calling you?
Don't

Common pitfalls this week

  • Skipping webhook signature verification — anyone can send a forged event to your endpoint.
  • Leaving workarounds undocumented — the next engineer will delete them and break the integration.
  • Testing against production data without idempotent calls and a clear rollback plan.
  • Treating 'it works on the happy path' as done — error handling and alerting are the real job.
  • Stopping when the clean API does not exist instead of finding the next viable integration pattern.
Recap

If you remember three things

  • Adapt to the customer's reality — their environment is a constraint, not a complaint to fix.
  • Defensive integration assumes the other side will break, and builds to survive it gracefully.
  • Every workaround needs a three-line note: what it does, why it was necessary, and what to watch.
