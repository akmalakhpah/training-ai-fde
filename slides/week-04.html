The production loop

How an LLM app actually works

  1. Input — user data, context, retrieved documents arrive from upstream.
  2. Prompt assembly — your code builds the full prompt from parts: role, task, context, examples.
  3. Model call — one HTTP request to the API; the model is a black box at this point.
  4. Parse — extract the structure the model was supposed to return.
  5. Validate — confirm the output actually matches the schema and types you defined.
  6. Act — write to a database, call another service, return to the user.
  7. Handle failure — retry, fallback, log, alert; design this before it happens.
Mental model

Where things go wrong

What you expected to be hard

  • Choosing the right model
  • Writing the initial prompt
  • Getting an API key and making the first call
  • Reading the raw response

Where bugs actually live

  • Prompt assembly with real, messy input
  • Output parsing on edge cases — text that lies about being JSON
  • Missing validation — wrong types, missing fields, null values
  • No fallback when the model returns something unusable
  • Ignoring cost until the bill arrives
Prompting as engineering

A prompt is code — treat it like code

Anatomy of a good prompt

  • Role — who the model is in this context
  • Task — the specific thing it must do
  • Context — what it needs to know to do it well
  • Format instruction — the exact shape of the output
  • Examples — one or two make the task concrete
  • Constraints — what to do on bad or ambiguous input

It also has all the properties of code

  • Inputs — variables you interpolate at runtime
  • Expected outputs — the schema you defined
  • Edge cases — test inputs designed to break it
  • Versions — v1, v2, with a changelog of what changed and why
  • Tests — a set of inputs you run every time you change it
Prompting as engineering · a real example

One Anki task, three prompt versions

v1 — a wish

Tag this question: "{question}"

Returns a sentence, sometimes the wrong shape. Breaks your parser on edge cases.

v2 — role + format

You are a question tagger. Classify the question below. Reply as JSON: {"topic": ..., "difficulty": ...} topic in: Math, Science, English, History. difficulty in: easy, medium, hard. "{question}"

v3 — + example

...same as v2, plus: Example: Q: "What is 7 x 8?" {"topic":"Math", "difficulty":"easy"} Now tag: "{question}"
Structured output

Never trust a model to format itself

  • Asking nicely for JSON is not a contract — it is a wish.
  • The model adds preamble sentences, wraps output in markdown, or puts the wrong type in a field.
  • Structured outputs pass a JSON schema to the API — the model is constrained at decoding time, not by instruction.
  • It literally cannot produce a token sequence that violates the schema you passed.
  • This is the single biggest reliability improvement in one line of code.
  • Never parse free text when you can constrain the shape.
Structured output · real code

A schema the model cannot break — on the Claude API

The schema (the contract)

SCHEMA = { "type": "object", "properties": { "topic": {"type": "string", "enum": ["Math", "Science", "English", "History"]}, "difficulty": {"type": "string", "enum": ["easy", "medium", "hard"]} }, "required": ["topic", "difficulty"], "additionalProperties": False }

The call (guaranteed JSON)

resp = client.messages.create( model="claude-haiku-4-5", max_tokens=128, messages=[{"role": "user", "content": prompt}], output_config={"format": { "type": "json_schema", "schema": SCHEMA}}, ) tag = json.loads(resp.content[0].text) # tag == {"topic": "Math", # "difficulty": "easy"}
Error handling

Design for the bad response, not the demo response

  • Non-determinism is real — the same prompt produces a distribution of outputs, not a fixed answer.
  • Retries — exponential backoff on transient API errors; feed the validation error back on schema failures so the model can self-correct.
  • Timeouts — set them; a model call can hang indefinitely without a deadline.
  • Fallbacks — always explicit: if unusable after retries, return a safe default and log the raw response clearly.
  • Logging — always log input tokens, output tokens, and the raw response when something goes wrong; you cannot debug what you did not record.
Error handling · what the model actually returns

Four ways "give me JSON" breaks

Looks fine, isn't

Here is the JSON: {"topic": "Math", ...} # preamble ```json {"topic": "Math", ...} # fenced ``` {"topic":"Math", "difficulty": 2} # number, not # the enum string

The sneaky one

{"topic": "Algebra", "difficulty": "medium"}
  • Valid JSON. Valid string. Wrong value — "Algebra" is not one of your four topics.
  • Structured outputs stop the first three. The schema enum rejects this one.
  • Your code falls back to "Other" and logs the raw response. That is your reliability check.
Streaming

When streaming changes the experience

Without streaming

  • User sees nothing until the full generation completes
  • A 3-second response feels like a frozen screen
  • Fine for short, structured API calls
  • Simpler code path

With streaming

  • Tokens arrive as they are generated
  • User sees progress immediately — perceived latency drops dramatically
  • Critical for long-form output: feedback, explanations, summaries
  • One flag on the API client; handle partial buffers carefully

Live build · ~20 min

Classification feature — end to end on the Claude API

  • Step 1 — bare prompt: classify by topic, show it work then wobble on ambiguous input.
  • Step 2 — add format instruction and one example to the prompt; watch output stabilise across calls.
  • Step 3 — switch to structured outputs with a JSON schema (topic enum, difficulty enum); free text is no longer possible.
  • Step 4 — add a cost counter: print tokens in, tokens out, and dollars per call after every invocation.
  • Step 5 — wrap the call in try/except with exponential backoff retry and an explicit safe fallback.
How you build · markdown + AI tools

Plan in markdown — the language your AI tools read

Write a plan.md first

  • .md is the AI lingua franca — CLAUDE.md briefs the agent, README.md briefs humans, plan.md is where you think.
  • Five sections for the Anki feature: Goal · Output schema · Prompt versions · Failure modes · Cost target.
  • Commit it. It is versionable, reviewable in the PR, and directly readable by Claude Code — the plan becomes the brief.
  • You write the thinking down once, in a format both you and the agent read.

Understand with Claude Cowork

  • Claude Cowork — Anthropic's collaborative workspace: a shared doc you and Claude edit together, not a throwaway chat.
  • Paste your feature's code and prompt; ask it to map them onto the seven-step loop, draft the schema, and list the inputs most likely to break it.
  • Co-edit the plan.md with it so the understanding becomes a kept artifact.
  • A chat you lose; a markdown plan in your repo you keep, version, and hand to the next tool.
Self-learn · 6–8 hrs

Four topics — all on your Anki feature

  1. Prompt design and iteration (1.5–2 hrs) — iterate your Anki feature's prompt across three versions; measure and document the output change at each step.
  2. Reliable structured output (1.5–2 hrs) — break the naive JSON path with three odd Anki inputs; then fix it with a schema and see what the schema catches.
  3. Token economics and cost (1–1.5 hrs) — measure cost per call for your feature from response.usage; multiply to a million calls and find the biggest cost lever.
  4. Handling non-determinism (1 hr) — call the same prompt five times; write the validation, retry, and fallback code that survives the variation.

Reliability check · every Phase 2 deliverable

Name one input that breaks it — and what catches it

  • Every Phase 2 deliverable carries a one-line reliability check in the submission.
  • Name one specific input that would make your feature produce wrong or malformed output.
  • State exactly what in your code catches it: schema validation, a retry, a fallback, a log line.
  • Vague answers fail: 'the try/except handles it' names nothing.
  • Precise answers pass: name the input, name the failure mode, name the catch.
Example: 'An empty string input causes the schema to reject the null topic field; caught by Pydantic validation, logged with the raw response, and returns the safe default {topic: unknown, difficulty: unknown}.'
Don't

Common pitfalls this week

  • Stopping at the first working call — the demo response is not the production response.
  • Parsing free-text JSON with a regex instead of using structured outputs.
  • Ignoring cost until the end — build the cost counter on call one, not after the feature is done.
  • Treating retry logic as a stretch goal — it is on the checklist and on the rubric.
  • Using Opus for a task that Haiku handles just as well at 5× the cost.
  • Writing a reliability check that names no specific input and no specific catch — it will not pass.
Recap

If you remember three things

  • The model is the easy part — prompt assembly, output parsing, validation, and failure handling are the job.
  • Structured outputs with a schema turn a wish for JSON into a guarantee enforced at decoding time.
  • Every call has a cost; compute it on call one and multiply it to scale before you commit to a design or a model tier.
