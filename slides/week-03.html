Week 03 · Phase 1 — Production Foundations

Deploy, Observe, Recover

Shipping is not the finish line. The job is keeping it alive and knowing the moment it is not.

The gap

The demo runs on your laptop. Production runs where you are not watching.

— the moment every new engineer finds out

  • Expose a working API endpoint and keep config and secrets out of the code.

Block 1 — What 'live' means

The anatomy of a running service

  • Build artifact — the thing that actually runs, not the source code.
  • Environment — infrastructure, configuration, and network wired together.
  • Configuration — behaviour that legitimately changes between environments.
  • Secrets — credentials and keys that must never enter source control.
  • Public endpoint — the address the outside world calls.
Block 2 — Containers and environments

The same thing, everywhere

Why containers

  • Package code, runtime, and dependencies together.
  • Runs identically on every machine and platform.
  • Solves 'works on my machine' permanently.
  • The image is the build artifact you ship.

Why separate environments

  • Dev — break things freely, no users affected.
  • Staging — mirror of prod for final checks before release.
  • Prod — real users; treat every change carefully.
  • A bug caught in staging is a near-miss, not an incident.
Block 3 — Configuration and secrets

The rule with no exceptions

  • Nothing secret in the code. Nothing environment-specific hardcoded.
  • Use environment variables for values that change between environments.
  • Use a secret store (AWS Secrets Manager, Cloudflare Secrets) for credentials and keys.
  • A secret in git history is a secret forever — treat it as already leaked the moment it is pushed.
  • Live: move one hardcoded value to an env var and redeploy — class watches the whole cycle.
Block 4 — Two paths to live

Pick one for the build

Path A · AWS

  • Managed compute: Lambda or App Runner.
  • Full AWS ecosystem — IAM, CloudWatch, Secrets Manager.
  • Better for stateful, heavy, or long-running workloads.
  • Deploy with the AWS CLI or console.
  • Pick this if you need more control or heavier compute.

Path B · Cloudflare Worker

  • Runs at the edge — 300+ locations, sub-millisecond cold start.
  • Deploy with the Wrangler CLI in a few minutes.
  • Secrets via wrangler secret put, no console required.
  • Best for lightweight, latency-sensitive, stateless APIs.
  • Pick this for global reach and fast iteration.
Block 5 — Observability

Logs vs metrics — know the difference

Logs

  • Timestamped record of what happened, in sequence.
  • Tell you the exact moment and chain of events.
  • Best for: diagnosing a specific failure or request.
  • Never log secrets, tokens, or personal data.

Metrics

  • Aggregated measurements over time: counts, rates, latencies.
  • Tell you how often and how bad a problem is.
  • Best for: spotting trends and triggering alerts.
  • A metric can catch the problem before a user reports it.
Live · ~14 min

The live incident

  • Instructor breaks the deployed service on purpose — bad config, missing env var, or thrown error.
  • Class diagnoses using only logs and metrics. Source code is off limits.
  • Work the recovery cycle: form a hypothesis, check the logs, fix, redeploy, confirm.
  • Key lesson: stay calm and read. The logs almost always told you already.
Block 6 — API design

What a clean endpoint looks like

  • Clear inputs — validate before your logic runs, never trust the network.
  • Clear outputs — a consistent shape every caller can rely on.
  • Sensible status codes — 200 means success, not 'we tried our best'.
  • Explicit errors — tell the caller what went wrong and why, without leaking internals.
  • This is the shape your project service will expose and what the assessment checks.
Self-learn (6–8 hours)

How to use your week

  1. Topic A (2 hrs) — Deploy the hello-world to your chosen platform and confirm the public URL from a browser.
  2. Topic B (1–1.5 hrs) — Add one clean endpoint: structured input, structured output, correct status codes.
  3. Topic C (1.5 hrs) — Trigger an error on purpose and find the root cause using only the logs.
  4. Topic D (1 hr) — Move every hardcoded value to an env var; confirm the service still runs cleanly.
Weekly project

Deploy, break, recover, document

  • Deploy a small service to AWS or Cloudflare with a public URL.
  • Expose at least one API endpoint with clean inputs, outputs, and status codes.
  • Keep config and secrets out of the code — use env vars or a secret store.
  • Break it intentionally: bad config, missing variable, or a thrown error.
  • Diagnose and fix using only logs and metrics — no re-reading source code first.
  • Write the incident report: timeline, symptom, diagnosis from logs, fix, confirmation.
Watch out

Common pitfalls this week

  • Hardcoding a secret 'just for now' — one push makes it permanent in git history.
  • Moving on before confirming the deployed service is actually healthy and reachable.
  • Skimming logs — slow down and read every line around the failure timestamp.
  • Attempting both deployment paths — pick one and go deep on it.
  • Guessing at the fix without first forming a hypothesis from the evidence in the logs.
Recap

If you remember three things

  • Shipping is the beginning, not the end — a live service needs watching.
  • Logs are your only reliable window into what a running service is actually doing.
  • Config discipline — no secrets in code, ever — is what makes deploys safe.
