Week 06 · Phase 2 — AI Application Engineering

MCP and Tool Calling

A model that can only talk is a chatbot. A model that can call tools is an FDE's building block.

Learning objectives

By Friday you can

  • Explain MCP as the standard protocol for connecting models to tools and data.
  • Use an existing MCP server and complete a real task through it.
  • Build a custom MCP server that exposes tools with strict input schemas.
  • Add a guardrail that blocks a clearly bad action and holds when provoked.
  • Use Conductor to run more than one agent in parallel on real work.
The problem

Before MCP: N models times M tools equals N times M custom integrations, all by hand.

— the integration tax MCP eliminates

What MCP provides

Before and after a common protocol

Before MCP

  • Each tool wired by hand into each app
  • N models times M tools integrations
  • Bespoke schemas, bespoke error handling
  • Fragile, hard to reuse, hard to test

With MCP

  • One common protocol: any model, any tool
  • Tools, resources, and prompts exposed cleanly
  • Build a server once, connect many models
  • MCP is to AI tools what USB was to peripherals
The mechanics

How a model actually calls a tool

  1. The model reads your tool definition: name, description, and input schema.
  2. It decides to call the tool and outputs a structured request with arguments.
  3. Your code intercepts the request and runs the real function.
  4. The result is returned to the model as a new message in context.
  5. The model continues reasoning — possibly calling more tools.
{ name: 'get_topic', description: 'Given a student question, return its topic and difficulty. description is a prompt the model reads.', inputSchema: { type: 'object', properties: { question: { type: 'string', description: 'The full text of the question.' } }, required: ['question'] } }
Live · ~12 min

Connecting to an existing MCP server

  • Install and configure the filesystem (or database) MCP server — no custom code.
  • Connect Claude to the server with one config change.
  • Give Claude a real task that requires reading or searching the filesystem.
  • Watch the tool call go out, see the result come back, see Claude continue.
  • Open the raw logs: show the exact JSON the model sent and received.
Build it yourself

Structure of a custom MCP server

  1. Define the tool: name, description the model will read, strict input schema.
  2. Validate the input before you do anything — reject bad input early and cleanly.
  3. Do the actual work inside the validated path only.
  4. Return a clean, structured result the model can reason about.
  5. Expose the server so any MCP-compatible client can connect to it.
from mcp.server.fastmcp import FastMCP mcp = FastMCP('classifier') @mcp.tool() def classify_question(question: str) -> dict: 'Given a student question, return its topic and difficulty.' if not question.strip(): raise ValueError('question must not be empty') return { 'topic': 'math', 'difficulty': 'medium' }
Live · ~16 min

Building and calling a custom MCP server

  • Start with the FastMCP scaffold — under 20 lines to a working server.
  • Define one tool with a strict input schema and a tight, specific description.
  • Run the server locally and connect Claude to it.
  • Give Claude a task that forces it to call your tool — watch the round trip end to end.
  • Change the description to something vague and show the model misfiring.
Chaining tools

Multi-step agent loops over MCP tools

  • The loop: call a tool, get result, decide next step, call another tool.
  • The model holds state in its context window between calls.
  • Your server stays stateless — the model is the orchestrator.
  • The loop ends when the model decides the task is complete.
  • Design each tool to return clean output the model can use as next-step input.

Safety by design

Why guardrails exist and where they live

  • A model with hands can do real damage: delete, send, charge, overwrite.
  • Prompts are not guardrails — a model can reason past a prompt instruction.
  • Guardrails live inside the tool function, before the action runs.
  • Design for the action you would be fired for: block it unconditionally.
  • A guardrail a clever prompt can bypass is not a guardrail.
def delete_records(table: str, where_clause: str) -> dict: if not where_clause.strip() or where_clause.strip() == '1=1': raise ValueError( 'Guardrail: refusing to delete all records. Provide a specific filter.') # proceed only after guard passes
Live · ~12 min

Guardrail: add it, then provoke it

  • Add a guardrail to the custom server tool — one check before the action.
  • Provoke it: send the exact bad input the guardrail must block.
  • Show it holding: the action does not run, a clean error is returned.
  • Show Claude receiving the rejection and responding appropriately.
  • Try to prompt-engineer around the guardrail and confirm it still holds.
More than one agent

Conductor and parallel agents — why now, not Week 1

  • Week 1: one task, one agent — parallelism would add confusion, not speed.
  • Week 6: the intern has tools, a server, tests, and refinement — real parallel work.
  • Conductor runs two or more agents simultaneously against the same tools.
  • Parallel agents help when tasks are independent and one can judge the other.
  • Avoid parallelism when tasks share state or one output feeds the next.

Self-learn this week

6 to 8 hours across four topics

  1. Topic A (1 hr): Read the MCP overview at modelcontextprotocol.io. Write three sentences explaining it to a colleague who has never used it.
  2. Topic B (1.5 hrs): Connect to an existing MCP server (filesystem or database) and complete one real task end to end. Follow the DataCamp MCP guide.
  3. Topic C (2 to 2.5 hrs): Build a custom FastMCP server — one tool, strict schema, called from Claude. Follow the Towards Data Science tutorial or the official build-server page.
  4. Topic D (1.5 hrs): Add a guardrail to your tool, provoke it, then run two agents in Conductor against your server and note whether it helped.
Weekly project

Build a custom MCP server for a real multi-step task

  • Expose at least two tools with strict input schemas through your MCP server.
  • Drive Claude through a real multi-step task using those tools in an agent loop.
  • Add at least one guardrail that blocks a clearly bad action and holds when provoked.
  • Use Conductor to build and test in parallel — include one line on whether it saved time.
  • This server is what Week 7 will automate — build it to be called by something other than you.
Reliability check

Name the bad input — then prove it is blocked

  • Name one specific bad input or action your guardrail must block. Be concrete, not vague.
  • Deliberately send that exact input to your running server.
  • Confirm the action does not execute and a clean error is returned.
  • Include this in your submission: one sentence naming the bad input, one confirming it held.
  • During the demo you will be asked to provoke it live — if it does not hold, the work is not done.
Reliability check (example): 'The guardrail blocks delete_records when where_clause is empty or equals "1=1". Confirmed: sending where_clause="" raises ValueError and zero rows are deleted.'
Don't

Common pitfalls this week

  • Writing a vague tool description — the model reads it as a prompt and calls the tool incorrectly.
  • Putting the guardrail in the system prompt instead of inside the tool function code.
  • Building a server that only works when you are watching it — hardcoded paths, local-only env vars.
  • Running parallel agents on tasks that share state — they will conflict and corrupt each other.
  • Skipping the reliability check — if you have not provoked the guardrail deliberately, you do not know it works.
Recap

If you remember three things

  • MCP is the USB of AI tools — one protocol so any model talks to any tool.
  • The guardrail lives inside the tool function, not inside the prompt.
  • Build the server to be called by something other than you — Week 7 depends on it.
