Block 1 · SQL

The constructs that matter most

Joins and aggregates

  • INNER JOIN — only matching rows on both sides
  • LEFT JOIN — all left rows, NULLs where right has no match
  • GROUP BY + COUNT / SUM / AVG
  • HAVING — filter after aggregation, not before

CTEs and subqueries

  • WITH cte_name AS (...) — give a step a name
  • Chain CTEs: each can reference the one above it
  • Subquery: the same idea, inline — harder to read
  • Rule of thumb: if you need to name it, use a CTE
Block 2 · Window functions

Calculate across rows without collapsing them

  • ROW_NUMBER() OVER (PARTITION BY key ORDER BY ts) — enumerate rows per group. Use this to find duplicate keys.
  • RANK() OVER (...) — like ROW_NUMBER but tied rows share a rank.
  • LAG(col, 1) OVER (ORDER BY ts) — pull the previous row's value into the current row. Use this to detect reversals or drops.
  • SUM(amount) OVER (ORDER BY ts) — running total without collapsing rows.
  • Key idea: OVER(...) means 'across this window of rows, keep every row intact'.
Live · ~15 min

Build a query from simple to complex

  • Start: SELECT with an INNER JOIN to answer one real question about the dataset.
  • Refactor with a CTE — show the same query before and after; ask which is clearer.
  • Add ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date) to surface duplicate rows per customer.
  • Add LAG(order_total) to flag orders where the total reversed or hit zero.
  • Ask trainees to predict the output before each run — keep it interactive.
Block 3 · Data quality

The usual suspects in every export

  • Nulls where there should not be — required fields that are empty in practice.
  • Duplicate keys — two rows that claim to be the same record with different values.
  • Wrong types — prices stored as text, dates as integers, booleans as 'Y'/'N'.
  • Out-of-range values — quantities of -1, ages of 200, timestamps in the future.
  • Inconsistent categories — 'MY', 'Malaysia', 'malaysia', 'MYS' all meaning the same thing.
  • Broken rows — partial imports, truncated text, misaligned columns from a CSV gone wrong.
Block 3 · Profiling

The profiling pass — run this first, always

SELECT 'order_id' AS col, COUNT(*) - COUNT(order_id) AS nulls, COUNT(DISTINCT order_id) AS distinct_vals FROM orders UNION ALL SELECT 'order_total', COUNT(*) - COUNT(order_total), COUNT(DISTINCT order_total) FROM orders UNION ALL SELECT 'customer_id', COUNT(*) - COUNT(customer_id), COUNT(DISTINCT customer_id) FROM orders;
  • Run this on every export before touching the data — catalogue every problem found.
  • A null count above 0 on a 'required' column is a confirmed bug, not a maybe.
Block 4 · Cleaning

One decision per problem

Your four options

  • Fix — correct the value if you are certain of the right answer
  • Drop — remove the row if it is unrecoverable; document why
  • Flag — add a column marking the problem; least invasive
  • Escalate — ask the data owner when you lack the context to decide

The golden rule

  • Never clean in place — write the result to a fresh table
  • Keep the raw data intact and untouched
  • Your cleaning must be reproducible from scratch
  • Confirm row counts: raw = clean + dropped
Live · ~15 min

Messy export → clean, queryable table

  • Load the raw export as-is into a staging table — no cleaning yet, nothing modified.
  • Run the profiling pass and narrate every problem found.
  • Write a cleaning CTE: CAST wrong types, ROW_NUMBER to deduplicate, CASE WHEN to normalise categories, flag column for missing required fields.
  • INSERT INTO clean_orders SELECT ... FROM cleaning_cte — one statement, no side effects.
  • Confirm: raw row count = clean rows + dropped rows. Show the raw table is still intact.
Block 5 · Performance

Indexing a slow query — before and after

38 ms

CREATE INDEX idx ON orders(customer_id);
  • Run EXPLAIN ANALYZE before and after — read the actual cost, not just the wall-clock time.
  • Index columns used in WHERE, JOIN ON, and ORDER BY clauses.
  • Indexes cost on every INSERT and UPDATE — index with evidence, not instinct.
Block 6 · Non-technical view

Gemini + Google Sheets for stakeholder output

  • Export one cleaned aggregate result to a Google Sheet — paste or use the Sheets connector.
  • Open Gemini inside the Sheet and ask: 'Summarise the key finding in two sentences for a non-engineer.'
  • Review and edit the output before sending it anywhere — the model drafts, you own it.
  • The customer often wants the answer in a sheet, not a database terminal.
  • You understand the data; Gemini helps you communicate it — the direction matters.
Block 7 · AI-assisted SQL

The model drafts. You verify.

  • Ask Claude or Gemini for a window-function query — be specific about the table name and the task.
  • Run it on the training dataset and inspect every output column.
  • Pull a 5-row sample with a known answer and check the query result against it by hand.
  • Verification prompt: 'Walk me through what this query does row by row on this sample: [paste 5 rows].'
  • A wrong query that runs is more dangerous than one that errors — SQL does not tell you when it is wrong.

This week's project

Detect · Fix · Load — and explain it

  • Detect: profile the messy export — catalogue every data-quality problem before touching anything.
  • Fix: clean with SQL — decide fix vs drop vs flag for each problem, justify every choice.
  • Load: INSERT INTO a fresh, queryable table — confirm row counts match before and after.
  • Write: one paragraph explaining what was wrong, how you detected it, and what you decided.
  • Keep the raw data intact — someone must be able to reproduce your clean table from scratch.
  • The export includes broken rows, wrong types, missing values, and duplicate keys — plus at least one problem with no clean answer you must choose and justify.
  • Deliverables: the cleaned dataset (or fresh table), the SQL (readable and commented), and the one-paragraph note.
  • Stretch: a validation query that fails loudly if the same problem appears in a future export.
  • Stretch: a Gemini + Sheets non-technical summary of one finding.

Watch out for

Common pitfalls this week

  • Trusting the export before profiling it — the schema says it is clean; the data says otherwise.
  • Cleaning in place — if you UPDATE the raw table, you cannot undo a wrong decision.
  • Running an AI-generated query without checking it on a 5-row sample first.
  • Fixing the first obvious problem and stopping — one bad column usually has company.
  • Writing the explanation after the SQL as an afterthought — the note is half the deliverable.
  • Escalating immediately without profiling — the data owner will ask 'what did you find?' first.
Recap

If you remember three things

  • Never trust an export — profile it first, every time, without exception.
  • Never clean in place — write to a fresh table so your work is reproducible.
  • The model drafts SQL; you verify it — a wrong query that runs is worse than one that errors.
